# 实体链接与知识对齐智能体 — 定义与设计

## 一、智能体概述

**课题编号**：课题 10 — 实体链接与知识对齐智能体（专项能力智能体）

**一句话定位**：把实体指称链接到知识库标准实体（消歧/标准化/NIL/共指），被归一智能体和清洗智能体按需调用。

**输入**：文本 + 已识别的实体指称（mention）+ 知识库（含标准实体、别名）  
**输出**：每个指称链接到的标准实体（标准全称 + 唯一 ID）；消歧与共指结果；NIL 判定；链接依据

**设计原则**：
1. 弱自主规划 — 主流程模板固化，确保结果可复现
2. 强流程约束 — 每个 Skill 有明确 Schema，状态机控制流转
3. 能用规则不用 LLM — BM25、别名匹配、NIL 阈值均为纯规则
4. 全程可追溯 — 原值→新值→依据，支持回放与回滚

## 三、能力清单（Skills）

智能体内部编排 5 个原子能力，每个 Skill 独立暴露为 HTTP 端点，可被流水线以注册制热插拔集成。

### 3.1 实体标准化（Skill #1）

**端点**：`POST /standardize`  
**实现类型**：纯规则（哈希表查找）  
**功能**：别名/简称/曾用名 → 标准全称 + 唯一 ID  
**性能**：O(1)，单次 <0.1ms  
**为什么不用大模型**：别名映射是确定的键值对查找，哈希表即可解决，用 LLM 是浪费。

### 3.2 上下文消歧（Skill #2）

**端点**：`POST /disambiguate`  
**实现类型**：规则（5 信号加权评分）  
**功能**：对多个候选实体，结合上下文区分同名异指  
**性能**：单次 <1ms（纯计算，无网络调用）  
**为什么不用大模型**：5 信号规则消歧器已达到 90.85% 准确率，LLM 仅作为 gap<0.01 的可选兜底。

**消歧信号**：

| 信号 | 权重 | 计算方式 |
|---|---|---|
| 名称匹配 | 0.15 | 精确匹配→1.0，别名匹配→0.95，包含→0.6~0.95 |
| 先验概率 | 0.15 | cbrt 压缩，中频实体差距最小化 |
| 名称完整度 | 0.09 | 全称优先于简称，过度细化惩罚 |
| 类型一致性 | 0.15 | PER/ORG/LOC/EVENT 硬过滤 |
| 领域匹配 | 0.10 | 2+3gram 区分词在全文中的命中密度 |

### 3.3 实体链接（Skill #3，组合 Skill）

**端点**：`POST /link`  
**实现类型**：规则流水线（5 级瀑布）  
**功能**：完整流程：标准化 → 消歧 → NIL 检测 → 追溯

| 步骤 | 置信度 | 触发条件 |
|---|---|---|
| 标准名精确匹配 | 1.0 | mention == standard_name |
| 别名匹配 + 消歧器 | 0.95 | mention 命中 aliases |
| 模糊匹配（包含关系） | 0.80 | 名称包含，长度差 ≤8 |
| BM25 全文检索 | ≤0.70 | jieba 分词 + BM25 打分 |
| NIL 判定 | — | 以上均未命中 |

### 3.4 NIL 检测（Skill #4）

**端点**：`POST /nil_check`  
**实现类型**：规则（候选检索 + 多信号融合）  
**功能**：正确识别知识库中不存在的实体  
**性能**：单次 <0.5ms  
**为什么不用大模型**：NIL 判定本质是"KB 中是否存在此 mention 的别名/标准名"，哈希表直接解决。

### 3.5 共指消解（Skill #5）

**端点**：`POST /coref`  
**实现类型**：规则（最近前序同类型回链）  
**功能**：代词/指代词回链到具体实体  
**性能**：单次 <0.5ms  
**为什么不用大模型**：体育/新闻等结构化文本中的代词回链可通过"最近前序同类型实体"规则解决，准确率 94.3%。

## 四、流水线编排

```
归一智能体/清洗智能体
         │
         ▼
┌──────────────────────────────────────────────┐
│  实体链接智能体 (Entity Linking Agent)        │
│                                              │
│  1. 接收 (text + mention + KB)               │
│  2. 按需调用 Skill：                         │
│     ├─ 标准化 (standardize)                  │
│     ├─ 消歧 (disambiguate)                  │
│     ├─ 链接 (link = standardize+disambiguate)│
│     ├─ NIL 检测 (nil_check)                 │
│     └─ 共指消解 (coref)                     │
│  3. 返回 (standard_name + id + trace)       │
└──────────────────────────────────────────────┘
```

## 五、可追溯设计

每次链接调用自动生成 `trace_id`，记录每一步的：

| 字段 | 说明 |
|---|---|
| step_name | 步骤名称，如"别名匹配+消歧" |
| original_value | 输入值，如"世锦赛" |
| new_value | 输出值，如"世界羽毛球锦标赛 (EVENT_0160)" |
| reason | 决策依据，如"从8个候选中选择，得分=0.705" |
| timestamp | 时间戳 |

追溯 API：

| 端点 | 功能 |
|---|---|
| `GET /trace/{id}` | 查看完整处理链路 |
| `POST /trace/{id}/replay` | 回放验证：重新执行并对比结果 |
| `POST /trace/{id}/rollback` | 回滚：查看原始 mention 状态 |

## 七、配置开关

```python
# elagent/config.py
# LLM 消歧配置（默认关闭，可私有化部署）
llm_enabled: bool = False
llm_model: str = "openai/gpt-oss-20b:free"
llm_base_url: str = "https://openrouter.ai/api/v1"
llm_score_gap: float = 0.01
```

LLM 仅作为规则消歧器的补充——当 top-2 得分差 < 0.01 时触发。所有异常自动降级回规则结果。

## 八、与开源/通用方案的区别

| 维度 | 本智能体 | 通用方案 |
|---|---|---|
| 实体识别 | 不研究（假设已识别） | NER + 链接一体 |
| 消歧 | 规则 5 信号 + 领域 2-gram | 通常依赖 BERT 语义 |
| NIL 检测 | 规则召回判定 | 多数系统不处理 NIL |
| 共指消解 | 规则回链 | 通常用深度学习模型 |
| 可追溯 | 每步留痕+回放 | 通用工具不提供 |
| LLM 依赖 | 可选，默认关闭 | 强依赖大模型 |
| 部署 | 纯 Python，CPU 可运行 | 通常需要 GPU |
